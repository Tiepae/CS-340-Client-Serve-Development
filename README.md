# CS-340-Client-Server-Development


When I think about writing maintainable, readable, and adaptable programs, I think about structure first. If the structure is messy, everything else becomes harder to manage. With the CRUD Python module, I made a conscious effort to separate the database logic from the dashboard itself. Instead of mixing MongoDB queries directly into the notebook, I created a separate module that handled Create, Read, Update, and Delete operations. That made the code easier to follow and easier to fix if something broke. If I needed to change how the database was accessed, I only had to adjust the CRUD file instead of digging through the entire dashboard.

Working this way helped more than I expected. It kept things organized, but it also made the project feel more real. It was not just an assignment anymore. It felt like I was building an actual system with different parts that each had a specific responsibility. I also realized that I could reuse that CRUD module in the future. If I build a web app, an API, or even another dashboard that needs MongoDB access, I already have a foundation. That is something I did not always think about before, writing code not just to work once, but to work again later.

When I approach a problem as a computer scientist, I try to slow down and understand what is actually being asked before jumping into coding. With the Grazioso Salvare project, I focused on getting the database connection and CRUD functionality solid first. If the data layer does not work, the dashboard will not work either. That was different from many of my previous classes where the framework was already built and I just had to fill in missing pieces. In this project, I had to think more about design and compatibility. I had to make sure the database, the queries, and the visual components all worked together smoothly. It required more independent thinking and planning than I was used to.

In the future, I would spend even more time planning the structure before writing code. Making sure the database design supports the client’s needs, testing queries independently, and thinking about performance earlier would all make the final system stronger. I am starting to see that writing code is only part of the job. The planning and structure behind it matter just as much.

Computer scientists do more than just write programs. They solve problems in a structured way. They take something messy or inefficient and turn it into something organized and functional. That matters because companies rely on systems that work correctly and consistently. In this case, the database and dashboard make it easier for Grazioso Salvare to search, filter, and visualize their animal data quickly. Instead of manually sorting through records, they can make faster decisions with accurate information in front of them. That kind of efficiency directly supports the organization’s mission. Projects like this show how technical skills can translate into real world impact, and that is what makes computer science meaningful to me.
